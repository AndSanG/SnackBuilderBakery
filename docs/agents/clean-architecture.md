# Clean Architecture

The architectural approach this project is built around. Like the development process guide, this is written to be language and framework agnostic: it describes the generic structure and rules, not the specifics of this codebase. For how these ideas map onto the actual modules here, see the project's module map and contracts.

Clean Architecture integrates several older approaches that share one objective, the separation of concerns through layers: Hexagonal Architecture (Ports and Adapters), Onion Architecture, Screaming Architecture, DCI, and BCE. They differ in detail but agree that a system is divided into at least one layer for business rules and another for interfaces, with the business rules independent of any external concern.

## The dependency rule

Source code dependencies point in one direction only: inward. Nothing in an inner circle knows anything about an outer circle. A name declared in an outer circle (a database, a web framework, a UI) must never appear in inner code: not a function, class, variable, or any other named entity.

The outer circles are mechanisms; the inner circles are policies. The further in you go, the higher level the code becomes. This applies to data as well as names: a data format produced in an outer circle, especially one generated by a framework, must not be used by an inner circle.

Concentric circles, from innermost to outermost:

```
  +------------------------------------------------+
  |  Frameworks and Drivers                        |
  |   +------------------------------------------+ |
  |   |  Interface Adapters                      | |
  |   |   +----------------------------------+   | |
  |   |   |  Application Business Rules      |   | |
  |   |   |   +--------------------------+   |   | |
  |   |   |   |  Enterprise Rules        |   |   | |
  |   |   |   |  (Entities)              |   |   | |
  |   |   |   +--------------------------+   |   | |
  |   |   +----------------------------------+   | |
  |   +------------------------------------------+ |
  +------------------------------------------------+

  dependencies point inward only
```

## The layers

### Entities (Enterprise Business Rules)

The innermost circle. Entities hold the rules and data that would be true for the business even if there were no application, no database, and no UI. An entity can be an object with methods or a set of data structures and functions; what matters is that it captures the most general, high-level rules, the ones reusable across many applications in an enterprise. In a single application, these are simply its core business objects. They are pure: no framework imports, no serialization concerns, no persistence annotations. They are the least likely to change when something external changes, and change only when the business rules themselves do.

### Use Cases (Application Business Rules)

The application-specific workflows. A use case orchestrates entities to carry out a single operation a user or system can request. It depends on entities and on abstract ports, never on concrete infrastructure. Each use case has a clear input, a result, and defined failure outcomes. Use cases change when the application's behavior changes, not when a database or framework changes.

### Interface Adapters

The translators. This layer converts data between the form most convenient for use cases and entities, and the form most convenient for external agencies. Controllers, presenters, request and response models (DTOs), repository implementations, and gateways live here. Nothing in this layer contains business rules; it only adapts.

### Frameworks and Drivers

The outermost circle: the web framework, the database, the runtime, external services. This is where all the details go, and generally you write little here beyond glue code that talks to the next circle inward. The web is a detail. The database is a detail. They are kept on the outside where they can do little harm, and wired to the rest of the system at a single composition point.

### More than four layers

The four circles are schematic, not a rule. A system may need more layers. Whatever the number, the dependency rule always holds: source code dependencies point inward, and the level of abstraction rises as you move in. The outermost circle is low-level concrete detail; the innermost is the most general, highest-level policy.

## Ports and adapters

Boundaries between layers, and between modules, are crossed through ports: abstract interfaces that describe an operation in terms the inner code understands. The inner code declares the port it needs; the outer code provides an adapter that implements it.

This is the practical form of the dependency rule. A use case that needs to read or store data does not depend on a database client. It depends on a repository port. The concrete repository, which does know about the database, implements that port and is injected at composition time. The arrow of source-code dependency points inward (the adapter depends on the port), even though the flow of control at runtime points outward (the use case calls the adapter).

A port is defined by its consumer, sized to exactly what the consumer needs, and no larger. A consumer should not depend on operations it never calls.

## Crossing boundaries

Data that crosses a boundary is a simple, isolated structure. Inner circles never receive an outer circle's types: no database row objects, no framework request objects, no ORM entities reach a use case or an entity.

- Inbound: a request is mapped from its transport shape (a DTO) into the inputs a use case accepts.
- Outbound: a use case result is mapped into a response shape for the caller.

The rule runs both ways. Just as an outer type must not be passed inward, an entity or a database row must not be passed outward across a boundary. Whatever crosses is a plain, isolated structure in the form most convenient for the inner circle: a simple struct, a data transfer object, or plain function arguments. Mapping happens in the interface adapter layer. Entities and use cases stay ignorant of how data is transported or stored.

## Dependency inversion

Wherever control flow needs to point outward (an inner use case must cause something to happen in an outer layer), invert the source-code dependency with an interface. The inner layer owns the abstraction; the outer layer conforms to it. This is what keeps the dependency rule intact at every place control would otherwise leak outward.

## What this buys

A system built this way has the properties Clean Architecture is meant to produce:

- **Independent of frameworks.** The architecture does not depend on a feature-laden library. Frameworks are tools the system uses, not constraints it is crammed into.
- **Testable.** Business rules are tested without the UI, database, web server, or any external element, with small fakes standing in for ports.
- **Independent of the UI.** The UI can change, or be replaced entirely (a web UI for a console UI), without changing the business rules.
- **Independent of the database.** The database can be swapped (one engine for another, SQL for a document store) without touching business rules.
- **Independent of any external agency.** The business rules simply do not know anything about the outside world.

Two consequences worth naming for day-to-day work:

- **Independent change.** Business rules change without dragging infrastructure along, and infrastructure changes without disturbing business rules.
- **A single place to understand wiring.** The full object graph is assembled at one composition point, so nothing else needs to know how the whole system fits together.

## Pragmatism: one implementation per abstraction

Clean Architecture is a means, not an end. Introduce a port, a strategy, or an extra layer when it solves a present problem: a real second implementation, a genuine boundary that needs to be testable, or a detail that must be kept replaceable. An abstraction with exactly one implementation and no near-term variation is usually premature. Keep the seam (a single well-named method, one boundary) and introduce the abstraction when its second implementation actually arrives.

The boundaries worth keeping even with one implementation today are the ones that exist for a concrete reason: keeping a use case testable without infrastructure, or isolating a detail that is explicitly planned to be replaced.

## Summary checklist

- Dependencies point inward only. No outer name appears in inner code.
- Entities are pure business rules, free of frameworks and persistence.
- Use cases orchestrate entities through ports, with clear inputs, results, and failures.
- Interface adapters translate; they hold no business rules.
- Frameworks, databases, and external services live at the edge behind ports, wired at one composition point.
- Ports are defined by their consumer and sized to its needs.
- Data crossing a boundary is a plain structure; inner code never sees outer types.
- Abstractions earn their place; one implementation is a seam, not yet an interface.
