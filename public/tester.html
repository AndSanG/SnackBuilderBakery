SnackBuilder endpoint tester

A scratch page to poke the bakery API by hand. Each button sends one HTTP request and dumps the raw response at the bottom. No login, no build step.

Typical run-through:
  1. Add menu items with POST /menu (a cookie, a pastry...).
  2. List the menu with GET /menu and copy an item's id.
  3. Place an order: paste that id into the items JSON, pick a source, POST /orders. You get back an orderId and total (in cents). The order starts AwaitingPayment.
  4. Confirm payment: put the orderId in the order id box and POST confirm. The order moves to InKitchen and you get an estimatedReadyTime.
  5. Advance the clock: nothing cooks on its own, so POST /orders/reconcile promotes any order whose estimate has passed to Ready.
  6. Track any order anytime with GET status.

Each use case below lists its cases from the testing strategy (docs/testing-strategy.md): happy paths and edge cases. Tick each off as you run it. Boxes are remembered in this browser.

Change this to point at a server running on another machine (e.g. http://192.168.x.x:3000).
Menu management

View Menu — GET /menu. Lists every item with name, category, and price. Empty bakery returns [].

Add Menu Item — POST /menu. Validates and creates an item, returns it with its new id. price is integer cents (250 = $2.50); category sets bake time (Cookie 5, Pastry 10, Bread 20 min). Bad data → 400.

Update Menu Item — PATCH /menu/:id (uses the name/category/price above). Remove Menu Item — DELETE /menu/:id (returns 204, no body). Both need a real id from View Menu; unknown id → 404.

Ordering

Place Order — POST /orders. Checks the items exist, prices the order, and creates it AwaitingPayment. Returns a ticket (orderId + total cents). source sets kitchen priority: Vip > AppDelivery > WalkIn. menuItemId must exist and quantity ≥ 1, else 400. Empty items → 400.

Confirm Payment — POST /orders/:id/confirm. Payment auto-succeeds: enqueues the items in the kitchen, moves the order to InKitchen, returns an estimatedReadyTime. Unknown order → 404; already paid → 409. Track Order — GET /orders/:id: current status anytime.

Reconcile Kitchen — POST /orders/reconcile. Nothing bakes on its own: this advances the clock so items whose bake time has elapsed finish and their orders flip to Ready. Call it after Confirm Payment to see progress.

Real clock here: an order only goes Ready after the bake time has truly passed. The automated e2e suite fast-forwards time instead, so for instant Ready prefer npm run test:e2e.
